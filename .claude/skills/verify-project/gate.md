# gate.md — verify-project user gates (G0→G4)

각 단계 끝의 **사용자 게이트**. 단일 확인/선택으로 진행. 대상이 단순·명확하면 게이트는
**fast-track**(요약만 보이고 자동 통과 제안). 응답은 한국어, 세로 `key: value`, 가로표 금지.

게이트는 *판정을 사용자에게 넘기는 것이 아니라*, 매니저가 근거와 함께 판정한 결과를 **확인받는** 지점이다.
"no"는 해당 단계를 재실행하거나 범위를 조정한다.

---

## G0 — 범위·표면 확인 (T0 끝)

보일 것:
- 대상: `<경로>` / 감지 표면: `web|api|cli|lib|tui`(근거 1줄)
- 정상 케이스 후보: N건(요약)
- 수행 단계: `--stages` 범위
- 산출 위치: `<out>`

확인 질문(예): "표면을 `web`으로 보고 정상케이스 N건으로 T1~T5를 진행할까요? (다른 표면/범위면 알려주세요)"
- no → 표면 강제(`--surface`)·케이스 가감·단계 축소 후 재확인.

---

## G1 — 코드검증 수용 (T1 끝)

보일 것:
- 점검 컴포넌트 수 / 일관성 불일치 건수
- 결함표 요약: CRITICAL n / HIGH n / MEDIUM n / LOW n (적대적 재검증으로 하향된 항목 명시)
- probe 테스트: 작성 n / 통과 n / 실패 n (실패는 환경갭 vs 결함 구분)

확인: "코드검증 결과를 수용하고 설치검증(T2)으로 갈까요?"
- no → 특정 컴포넌트 재점검 또는 결함 재검증.

---

## G2 — 가동 확인 (T2 끝)

보일 것:
- 기동 컴포넌트/포트/PID, 헬스 결과(매니저 독립확인)
- 환경 갭(있으면): 항목 + 플랫폼버그 아님 근거
- 재기동 명령 요약

확인: "플랫폼이 떴고 헬스 통과했습니다. 기능검증(T3)으로 갈까요?"
- no / 기동 불가 → 갭 해소 시도 또는 범위 축소(T1 + 정적 점검만)로 보고.

---

## G3 — 정상 기능 확인 (T3 끝)

보일 것:
- 케이스 결과: PASS m / FAIL k (FAIL은 케이스명 + 원인 1줄)
- 증거: 스크린샷 n매(web) 또는 응답 캡처 위치

확인: "정상 케이스 m/총 PASS. 적대적·heavy 검증(T4)으로 갈까요?"
- no → 실패 케이스 재현·원인 분석 후 재확인.

---

## G4 — 적대적 검증 수용 (T4 끝)

보일 것:
- 적대적/heavy 케이스: 안전처리 s / 취약 v (취약은 케이스 + 영향)
- 사후 무결성: 컴포넌트 헬스 / 데이터 무결성 / orphan 자원
- 무음 상한 있었으면 명시

확인: "적대적 검증 완료(안전처리 s/총). 종합보고서(T5)를 작성할까요?"
- no → 취약 케이스 추가 재현 또는 케이스 보강.

---

## Fast-track 규칙

- 대상이 작고(단일 컴포넌트·명확한 표면) 결과가 명백하면, 각 게이트는 **요약 + "이대로 진행합니다"** 한 줄로 통과 제안하고 즉시 다음 단계.
- 단, **결함이 CRITICAL/HIGH**이거나 **기동 불가**이거나 **취약점 발견** 시엔 fast-track 금지 — 반드시 사용자 확인.
